TypeScript has a feature that, once understood, changes how you model data for good. Discriminated unions let the type system track exactly which shape of data you're working with at any point, eliminating entire categories of runtime bugs.

## The problem they solve

Consider a typical async state model:

```typescript
interface State {
  loading: boolean
  data?: User
  error?: Error
}
```

This is technically valid TypeScript but it lies to you. Nothing prevents a state where `loading` is `true` and `data` is set simultaneously. Or where `error` is set but `loading` is still `true`. You're forced to write defensive checks everywhere and hope for the best.

## Enter discriminated unions

```typescript
type State =
  | { status: 'idle' }
  | { status: 'loading' }
  | { status: 'success'; data: User }
  | { status: 'error'; error: Error }
```

The `status` field is the **discriminant** — a literal type that narrows the union in switch statements and conditionals.

```typescript
function render(state: State) {
  switch (state.status) {
    case 'idle':
      return <EmptyState />
    case 'loading':
      return <Spinner />
    case 'success':
      return <UserCard user={state.data} /> // data is guaranteed to exist
    case 'error':
      return <ErrorMessage error={state.error} /> // error is guaranteed to exist
  }
}
```

TypeScript narrows the type in each branch. Inside `case 'success'`, the compiler knows `state.data` exists. No optional chaining, no non-null assertions, no runtime surprises.

## Exhaustiveness checking

The real power comes with exhaustiveness checking:

```typescript
function assertNever(value: never): never {
  throw new Error(`Unhandled case: ${JSON.stringify(value)}`)
}

function render(state: State) {
  switch (state.status) {
    case 'idle': return <EmptyState />
    case 'loading': return <Spinner />
    case 'success': return <UserCard user={state.data} />
    case 'error': return <ErrorMessage error={state.error} />
    default: return assertNever(state)
  }
}
```

If you add a new variant to `State`, the `default` branch becomes a compile error. The compiler forces you to handle every case.

## Where to apply this

Discriminated unions shine in a few key patterns:

**API response types**

```typescript
type ApiResponse<T> =
  | { ok: true; data: T }
  | { ok: false; error: string; code: number }
```

**Event systems**

```typescript
type AppEvent =
  | { type: 'user.login'; userId: string }
  | { type: 'user.logout' }
  | { type: 'post.created'; postId: string; authorId: string }
```

**Form state**

```typescript
type FormState =
  | { phase: 'editing'; values: FormValues; isDirty: boolean }
  | { phase: 'submitting'; values: FormValues }
  | { phase: 'submitted'; result: SubmitResult }
```

## The broader principle

Discriminated unions are an instance of making illegal states unrepresentable. You encode constraints in the type system so the compiler enforces them, rather than relying on runtime checks and defensive programming.

Once you start thinking this way, you'll find opportunities for discriminated unions everywhere. The result is code that communicates intent more clearly and fails loudly at compile time rather than silently at runtime.
